  NODE 1 OF 4:   ARDUINO UNO Q   (The Physical Sensing Device in the Car)
════════════════════════════════════════════════════════════════════════════════
  What it does: Reads all battery sensors every 100ms and sends the data
                to the Snapdragon PC over Wi-Fi. Also shows warnings on an
                LED screen when the AI detects a problem.
────────────────────────────────────────────────────────────────────────────────

[ PHYSICAL SENSORS ]  (These are physical hardware probes touching the battery)
   ├── Cell 1 Voltage  (Analog Pin A0) — Measures battery cell 1 voltage
   ├── Cell 2 Voltage  (Analog Pin A1) — Measures battery cell 2 voltage
   ├── Cell 3 Voltage  (Analog Pin A2) — Measures battery cell 3 voltage  ⚠
   ├── Cell 4 Voltage  (Analog Pin A3) — Measures battery cell 4 voltage
   ├── Current Sensor  (Analog Pin A4) — Measures how many Amps flowing
   ├── Temperature     (Analog Pin A5) — Measures how hot the battery is
   ├── Gas Sensor MQ-2 (Analog Pin A6) — Detects dangerous gas leaks (PPM)
   └── Vibration MPU6050 (I2C Bus)    — Detects shocks and vibration (G force)
            │                    │
            ▼                    ▼
   [ Zephyr ADC Driver ]   [ Zephyr I2C Driver ]
   Converts raw analog      Reads digital vibration
   voltage → digital        X, Y, Z values over
   number (0 to 4095)       400kHz serial bus
            │                    │
            └─────────┬──────────┘
                      │   Raw digital sensor numbers
                      ▼
         ┌────────────────────────────────────────┐
         │   ZEPHYR TASK THREAD 1                 │
         │   (Wakes up every 100ms = 10 times/sec)│
         │                                        │
         │   1. ADC reads all 7 analog channels   │
         │   2. I2C reads MPU6050 vibration data  │
         │   3. Converts numbers to real units:   │
         │        ADC raw 2912  →  3.82 Volts     │
         │        ADC raw 1515  →  34.5 Celsius   │
         │        ADC raw  623  →  1.25 Volts ⚠  │
         │        ADC raw  880  → -12.4 Amps      │
         │        ADC raw   47  →   12 PPM gas    │
         └───────────────────┬────────────────────┘
                             │   Clean physical values
                             ▼
         ┌────────────────────────────────────────┐
         │   TINYML / DSP NOISE FILTER            │
         │   (The "Noise Eraser")                 │
         │                                        │
         │   WHY: The electric motor creates      │
         │   electromagnetic interference which   │
         │   makes sensor readings jump or spike  │
         │   randomly — even if nothing is wrong. │
         │                                        │
         │   HOW: Runs a 5-tap moving median      │
         │   filter across the last 5 readings.   │
         │   It keeps the middle (median) value   │
         │   and throws away spikes.              │
         │                                        │
         │   RESULT: Smooth, reliable readings.   │
         │   No false alarms from road bumps.     │
         └───────────────────┬────────────────────┘
                             │   Stable, filtered values
                             ▼
         ┌────────────────────────────────────────┐
         │   ZEPHYR TASK THREAD 2                 │
         │   (Bridge Writer)                      │
         │                                        │
         │   Formats values as text strings and   │
         │   writes them into shared memory:      │
         │                                        │
         │   Bridge.put("voltages",               │
         │       "3.82, 3.81, 1.25, 3.83")        │
         │   Bridge.put("temperature",  "34.2")   │
         │   Bridge.put("current",     "-12.4")   │
         │   Bridge.put("vibration",    "0.08")   │
         │   Bridge.put("gas",            "12")   │
         │                                        │
         │   Think of it as writing sticky notes  │
         │   on a shared whiteboard that the      │
         │   Linux side can read at any time.     │
         └───────────────────┬────────────────────┘
                             │   Values now in shared SRAM memory
                             ▼
         ┌────────────────────────────────────────┐
         │   RPC INTER-CORE BRIDGE                │
         │   (The shared whiteboard / memory)     │
         │                                        │
         │   A hardware shared memory table that  │
         │   both the STM32 chip (Zephyr) and     │
         │   the QRB2210 Linux chip can read and  │
         │   write simultaneously.                │
         │                                        │
         │   Mapped variables:                    │
         │   "voltages"     → "3.82,3.81,1.25,3.83"

         │   STEP 1: Read the shared whiteboard   │

          │                           │
  (Wi-Fi Connected)           (Wi-Fi Disconnected)
          │                           │
          ▼                           ▼
 ┌─────────────────────┐    ┌──────────────────────────┐

                        │   Feedback now in shared memory

         │   ZEPHYR TASK THREAD 3                 │

  QUICK REFERENCE:  HOW DATA TRAVELS END-TO-END IN ONE SENTENCE PER STEP
================================================================================

  STEP  1: Battery sensors measure voltage, heat, gas, and vibration.
  STEP  2: ADC/I2C drivers convert those measurements to digital numbers.
  STEP  3: Zephyr Thread 1 reads numbers and converts to real physical units.
  STEP  4: Noise filter removes EMI spikes caused by the electric motor.
  STEP  5: Bridge.put() writes clean values into shared memory (whiteboard).
  STEP  6: Linux gateway reads the whiteboard, wraps values into JSON.
  STEP  7: If Wi-Fi is up → MQTT publish; if down → save to eMMC cache.
  STEP  8: Snapdragon PC broker receives JSON → backend.py extracts 8 values.
  STEP  9: Hexagon NPU runs Trust Model → Cell 3 is 14% trusted (FAULT!).
  STEP 10: Hexagon NPU runs SOH Model  → Battery is 94.2% healthy overall.
  STEP 11: Decision: Trust < 30% → publish "FAULT" to MQTT.
  STEP 12: Arduino gateway receives "FAULT" → writes to bridge memory.
  STEP 13: STM32 LED thread reads "FAULT" → flashes "!" on 8×13 LED matrix.
  STEP 14: WebSocket server broadcasts all data to dashboard and phone.
  STEP 15: 3D Dashboard turns Cell 3 red; phone app sounds alarm.
  STEP 16: User asks "Why is Cell 3 faulty?" to offline Llama-3 copilot.
  STEP 17: Llama-3 reads last 1,000 sensor readings and replies clearly.
  STEP 18: At midnight, Snapdragon PC uploads today's summary to Cloud.
  STEP 19: Cloud retrains smarter AI models on fleet-wide fault data.
  STEP 20: New AI model pushed back to all cars silently via OTA update.
