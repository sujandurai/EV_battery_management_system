+==============================================================================+
|  PART 4:  DATA ANALYSIS LAYER  (Understanding the Data)                     |
+==============================================================================+


                  +------------------------------------------+
                  |  FLEET ANALYTICS ENGINE                  |
                  |                                          |
                  |  Runs scheduled analytics jobs on the   |
                  |  database to find patterns.             |
                  |                                          |
                  |  JOB 1: SOH Degradation Curve           |
                  |  Plots SOH% vs charge cycle count for   |
                  |  every vehicle model. Detects vehicles  |
                  |  degrading faster than expected.        |
                  |                                          |
                  |  JOB 2: Fault Pattern Clustering        |
                  |  Groups fault events by cell number,    |
                  |  vehicle model, and temperature range.  |
                  |  Example finding:                       |
                  |  "Cell 3 disconnect appears 6x more     |
                  |   often in vehicles built in batch      |
                  |   B-2024-07 -- likely a connector       |
                  |   quality issue from that factory run." |
                  |                                          |
                  |  JOB 3: Thermal Risk Scoring            |
                  |  Identifies vehicles with temperatures  |
                  |  repeatedly above 45C -- flags them     |
                  |  for early battery health review.       |
                  |                                          |
                  |  JOB 4: Labelling for ML Training       |
                  |  Tags each incident as:                 |
                  |  - REAL_FAULT (actual cell problem)     |
                  |  - SENSOR_NOISE (false EMI reading)     |
                  |  - LOOSE_WIRE (connector disconnect)    |
                  |  - THERMAL_RUNAWAY (real overheating)   |
                  |                                          |
                  |  This labeled data becomes the training |
                  |  dataset for the model retraining step. |

+==============================================================================+
|  PART 6:  MODEL COMPILATION  (Preparing for the Car's NPU Chip)             |
+==============================================================================+


                  +------------------------------------------+
                  |  QUALCOMM AI HUB COMPILER                |
                  |                                          |
                  |  The generic ONNX model file must be     |
                  |  compiled into a QNN (Qualcomm Neural    |
                  |  Network) file to run on the Hexagon NPU |
                  |  inside the Snapdragon X Elite chip.     |
                  |                                          |
                  |  WHY CAN'T IT RUN ONNX DIRECTLY?        |
                  |  The Hexagon NPU has a custom hardware   |
                  |  instruction set. The QNN compiler       |
                  |  translates the model operations into    |
                  |  Hexagon-native instructions for maximum |
                  |  speed and minimum power usage.          |
                  |                                          |
                  |  COMPILATION COMMAND:                    |
                  |                                          |
                  |  qai-hub-client.compile(                 |
                  |    model  = "sensor_trust_v2.onnx",      |
                  |    device = "Snapdragon X Elite",        |
                  |    input_specs = {                       |
                  |      "telemetry_vector": [1, 8]          |
                  |    },                                    |
                  |    options = "--target_runtime qnn       |
                  |               --quantize int8"          |
                  |  )                                       |
                  |                                          |
                  |  WHAT THE COMPILER DOES:                 |
                  |  1. Parses the ONNX model graph          |
                  |  2. Fuses operations (e.g. Conv+ReLU)   |
                  |     into single NPU instructions        |
                  |  3. Allocates tensors to fastest         |
                  |     Hexagon memory banks (VTCM)          |
                  |  4. Quantizes weights to INT8 precision  |
                  |  5. Runs test inference to verify output |
                  |  6. Saves output as .qnn binary file     |
                  |                                          |
                  |  OUTPUT FILES:                           |
                  |  - sensor_trust_v2.qnn   (Trust Model)  |
                  |  - soh_estimator_v2.qnn  (SOH Model)    |
                  |                                          |
                  |  Size:   ~4 MB each (INT8 compressed)   |
                  |  Latency: 1.2ms on Hexagon HTP          |
                  +------------------+-----------------------+
                                     |
                                     | (compiled .qnn files ready)
                                     |
                                     v



+==============================================================================+
|  PART 7:  OTA DELIVERY  (Pushing New AI to Every Car Silently)              |
+==============================================================================+


                  +------------------------------------------+
                  |  OTA UPDATE DELIVERY SERVER              |
                  |                                          |
                  |  Manages secure distribution of new      |
                  |  model files to all registered vehicles. |
                  |                                          |
                  |  STEP 1: VERSION REGISTRY CHECK          |
                  |  Checks which model version each         |
                  |  Snapdragon PC currently has.            |
                  |  Database table: model_versions          |
                  |  Example:                               |
                  |    V-001: trust_v1.qnn  <-- old         |
                  |    V-002: trust_v1.qnn  <-- old         |
                  |    V-003: trust_v2.qnn  <-- already new |
                  |                                          |
                  |  STEP 2: NOTIFY OUTDATED VEHICLES        |
                  |  Sends a push notification to V-001 and  |
                  |  V-002: "Model update available."        |
                  |                                          |
                  |  STEP 3: SECURE FILE TRANSFER            |
                  |  When Snapdragon PC is online and idle:  |
                  |  - Downloads sensor_trust_v2.qnn         |
                  |  - Downloads soh_estimator_v2.qnn        |
                  |  - Verifies file checksum (MD5/SHA256)   |
                  |    to ensure file is not corrupted.      |
                  |                                          |
                  |  STEP 4: ATOMIC MODEL SWAP               |
                  |  backend.py detects new model files.     |
                  |  Waits for the current 100ms inference   |
                  |  cycle to finish.                        |
                  |  Closes the old ONNX Runtime session.    |
                  |  Opens a new session with new .qnn file. |
                  |  If it fails, rolls back to old model.  |
                  |                                          |
                  |  STEP 5: CONFIRM UPDATE                  |
                  |  Snapdragon PC sends confirmation to     |
                  |  cloud: "V-001 now running trust_v2.qnn" |
                  |  Cloud updates the model_versions table. |
                  +------------------+-----------------------+
                                     |
                                     v

                          |   NPU Trust Engine  |
                          |   updated silently. |
                          |   Accuracy: 95.3%"  |
